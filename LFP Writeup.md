# Navvy — Lo-Fi Prototype Write-Up

**Aryaman Goenka, Krish Kapoor, Aaron Li**
**EECS 497 — Winter 2026**

## Requirements Addressed by the LFP

Our Lo-Fi Prototype addresses several core requirements from our project specification. First, we implemented **AI-powered itinerary generation**: users can type a natural language prompt (e.g., "Plan a date night in Ann Arbor") and receive a structured 3–5 activity itinerary generated by an LLM. Each activity includes a name, description, category, time estimate, and real-world address. This directly satisfies our highest-priority requirement of natural language activity planning.

Second, we built a **map-based display** of generated itineraries using Leaflet and OpenStreetMap. Activities appear as numbered pins on an interactive map, with a scrollable card list below. This addresses the map + search interface requirement from our scope definition.

Third, we implemented a **user authentication flow** with sign-up and login pages that protect the dashboard behind a session. While currently cookie-based for simplicity, the flow mirrors the full auth experience we will ship in the final product.

Finally, we scaffolded the **four-tab navigation structure** (Plan, Feed, Community, Profile) that reflects our app's intended information architecture. The Feed and Community tabs contain static placeholder content showing what shared experiences and group planning will look like, and the Profile tab includes disabled fields for user preferences (allergies, dietary restrictions, budget, walking distance).

## Plan to Transform the LFP into the Final Project

Several major features remain for the final implementation:

**Agentic Booking:** Our core vision is an all-in-one concierge that doesn't just suggest activities but can actually book them. We plan to integrate reservation APIs (OpenTable/Resy) so users can confirm an entire itinerary with a single tap — turning Navvy from a planner into a true agent.

**Persistent User Memory:** We want Navvy to learn user preferences over time. If a user mentions a peanut allergy or indicates they dislike certain types of venues, that information should be stored and automatically applied to all future itinerary generation — no need to re-specify constraints each session.

**Social & Group Features:** The Feed and Community tabs need to become fully functional. This includes real shared itineraries between friends, the ability to copy or join others' plans, and group itinerary coordination where multiple users can collaboratively build and edit a shared plan.

**Infrastructure Upgrades:** We plan to migrate the map to Google Maps for richer place data and better UX, switch to a more capable LLM (Gemini) for higher-quality recommendations, and implement real authentication with Supabase for persistent user accounts and data storage.

**Profile & Preferences:** The currently disabled preference fields will become functional inputs that feed directly into the LLM's context, ensuring generated itineraries respect dietary restrictions, budget limits, and mobility constraints.
